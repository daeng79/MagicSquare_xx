# 역할 분리 Mom Test 시뮬레이션 보고서 — MagicSquare_xx

## 1. 문서 목적

- MagicSquare_xx 프로젝트의 **문제 인식**을 Rob Fitzpatrick *The Mom Test* 방식으로 검증하기 위한 **역할 분리 3턴 시뮬레이션** 결과를 정리한다.
- 미래 의향·솔루션 찬성이 아닌 **과거 사실·행동·손실**만을 근거로 진짜 문제를 도출한다.
- 본 문서는 **가상 인터뷰**이며, 실제 응답자·실제 경험과 무관하다.

## 2. 시뮬레이션 개요

| 항목 | 내용 |
|---|---|
| 진행 방식 | 역할 분리 Mom Test 3턴 (질문 → 답 → 추궁 → 답 → 추궁 → 답) |
| Role 1 | 인터뷰어 — Mom Test 규칙 준수, **질문만** |
| Role 2 | 페르소나 — 프로그래밍 수업에서 부분 마방진 과제를 받은 학생 |
| 금지 질문 | "만들면 좋겠어?", "이 기능 필요해?", "TDD 하면 어때?" |
| 허용 질문 | "마지막으로 ~했을 때", "그때 몇 분 걸렸어?", "뭐 때문에 포기했어?" |
| 라벨 | ⚠️ **시뮬레이션** — plausible한 가상 과거 사실만 사용 |

## 3. 페르소나 (가상)

- **2학년 `자료구조 기초` 수업** 수강생
- 과제: 4×4 격자에 빈칸 2개가 있는 부분 마방진을 읽어, 나머지 숫자를 채우고 **행·열·대각선 합이 34**인지 검증하는 프로그램 작성
- 제출 기한 1주일

## 4. 인터뷰 전문 (Q&A)

### Turn 1 — Q1. 마감 전날 진행 과정

**Q.** 마지막으로 프로그래밍 수업에서 **부분 마방진 과제**를 받았을 때, **제출 마감 전날** 그 과제를 어떻게 진행했는지 — 처음 연 순간부터 막히기 직전까지 — 기억나는 대로만 말해 주세요.

**A.** 수업에서 받은 예시 격자 파일을 열고, 빈칸 좌표를 출력해 보는 코드부터 짰어요. 빈칸이 (2,3)이랑 (3,1)이었고, 1~16 중 아직 안 쓴 숫자는 7이랑 11이었어요. 일단 7과 11을 넣어서 행 합이 34인지 확인했는데, 세 행은 다 맞았어요. 그다음 열 합도 맞아서 "거의 끝났다"고 생각했고, 대각선은 아직 안 봤어요. 그 상태에서 약 25분 지나서 잠깐 쉬려고 파일을 닫았어요.

### Turn 2 — Q2. (추궁) 대각선 확인을 미룬 이유

**Q.** "거의 끝났다"고 느꼈을 때, **대각선은 아직 안 봤다**고 했는데 — 그때 **왜** 대각선 확인을 미뤘나요? 당시에 머릿속에서 어떤 순서로 검증하려고 했는지 기억나요?

**A.** 교수님이 수업에서 "행·열부터 맞추면 대부분 된다"고 하셔서, 행·열이 맞으면 거의 정답이라고 생각했어요. 대각선은 "마지막에 한 번만 보면 되겠지"라고 여겼고, 손으로 계산하는 것도 귀찮아서 코드로 나중에 돌리려고 했어요. 검증 순서를 종이에 적어 두지는 않았어요. 머릿속 순서는 **행 → 열 → (나중에) 대각선**이었어요.

### Turn 3 — Q3. (추궁) 마감 당일 결과

**Q.** 그다음 날 — 즉 **마감 당일** — 그 파일을 다시 열었을 때, 실제로 **무엇을 했고**, 그 결과 **과제는 어떻게 끝났나요**?

**A.** 다음 날 다시 열어서 대각선 합을 손으로 계산했는데, 주대각선이 34가 아니었어요. 38이었어요. 7과 11 위치를 바꿔 봤는데 이번엔 행 하나가 34가 아니었어요. 어떤 조합이 맞는지 체계적으로 적어 보지 않고, 숫자 두 개만 바꿔 가며 15분 정도 시도했어요. 결국 "어느 제약이 깨졌는지"를 정리하지 못한 채, 예시 출력 형식만 맞춘 **부분 구현 코드**를 제출했어요. 빈칸 채우기 로직은 넣지 않았고, 자동 채점에서 **기능 미구현**으로 감점됐어요. 비슷한 유형 과제가 다음 주에 또 나왔는데, 그때는 **과제를 열지 않고** 조교에게 "다른 팀원이 하기로 했다"고 말했어요. (실제로는 혼자 과제였지만 솔직히 하기 싫었어요.)

## 5. 확인된 과거 사실 (가상 증거)

| # | 과거 사실 | 신호 강도 |
|---|---|---|
| 1 | 행·열만 맞으면 "거의 끝"이라고 판단 | 착각 패턴 |
| 2 | 대각선 검증을 의도적으로 미룸 | 검증 순서·우선순위 미고정 |
| 3 | 막혔을 때 조합을 체계 없이 2개만 바꿔 봄 | 다음 행동 불명 |
| 4 | 부분 구현만 제출 → 감점 | 미완료 제출 (시간·점수 손실) |
| 5 | 다음 유사 과제를 열지 않음 | 학습 회피 패턴 |
| 6 | **25분 + 15분** 소요 | 비용(시간) |

## 7. 문제 정의

### 7.1 표면 문제 (잘못된 정의 — 솔루션 섞임, 금지)

- "4×4 부분 마방진 **정답을 자동으로 채워주는 프로그램**이 필요하다."
- "**대각선 검증 함수**를 과제 템플릿에 넣어 주면 된다."
- "**TDD·pytest**로 마방진 솔버를 만들면 학생이 안 막힌다."

### 7.2 진짜 문제 (한 문장)

**여러 제약(행·열·대각선)이 동시에 걸린 과제에서, 중간에 "어디까지 맞고 어디가 틀렸는지"를 순서 있게 판별하지 못하면 학생은 "거의 맞음" 착각 후 시간을 쓰고도 제출을 미완료로 끝내고, 같은 유형 과제를 다시 열지 않게 된다.**

### 7.3 문제 분해

| 관찰 | 해석 |
|---|---|
| 행·열만 맞으면 "거의 끝" | **부분 만족**을 **전체 만족**으로 오인 |
| 대각선 확인 미룸 | 검증 **순서·우선순위**가 고정되지 않음 |
| 조합을 체계 없이 시도 | **다음 확인 단계**를 모름 |
| 부분 구현 제출·감점 | 판단 체계 부재가 **학습·성과 손실**로 이어짐 |
| 유사 과제 회피 | 일회성 좌절이 **학습 중단 패턴**으로 고정됨 |

## 8. 기존 보고서와의 정합성

Report/06(실사용자 Mom Test 인터뷰)과 본 시뮬레이션은 동일 축의 문제를 가리킨다.

| Report/06 개념 | 본 시뮬레이션 증거 |
|---|---|
| "거의 맞음" 착시 | 행·열 OK → 대각선에서 38 발견 |
| 판단 기준의 재현성 | 40분 후 미완료 제출·회피 |
| 실패를 구조적 신호로 | "합이 안 맞음"만 알고 **어느 제약 위반인지** 추적 못 함 |
| 조건 중심 문제 정의 | "정답 찾기"가 아니라 **막혔을 때 판단 체계 부재**가 핵심 |

**차이점:** Report/06은 ECB·TDD 학습자 맥락, 본 시뮬레이션은 **프로그래밍 수업 과제** 맥락. 진짜 문제의 본질은 동일하다.

## 9. MagicSquare_xx 프로젝트 함의 (문제만, 솔루션 제외)

- 프로젝트가 다루는 핵심은 **"빈칸 두 칸의 정답"** 자체가 아니다.
- **막혔을 때 어디가 깨졌는지를 구조적으로 드러내지 못하면**, 학습자는 시간을 쓰고도 미완료로 끝내거나 같은 유형을 회피한다.
- 요구사항·테스트·실패 조건은 **"정답 반환"**과 동등하게 **"위반 조건의 명시적 드러냄"**을 다루어야 한다.

## 10. Mom Test 워크북 (시뮬레이션 최종본)

### 페르소나

프로그래밍 수업에서 4×4 부분 마방진(빈칸 2개) 과제를 받은 학생. 행·열 검증 후 **대각선·조합 판단**에서 어려움을 겪는다.

### 인터뷰 답변 (과거 사실 — 가상)

1. 행·열만 맞으면 **"거의 끝"**이라고 판단했다.
2. **대각선 검증을 미뤘다** (행 → 열 → 나중에 대각선).
3. 마감 당일 주대각선 합이 **38**이라 막혔다.
4. 조합을 체계 없이 시도 후 **부분 구현만 제출** → 감점.
5. 다음 유사 과제를 **열지 않았다**.
6. **40분**(25+15) 소요.

### 표면 문제 (금지 — 솔루션 섞인 문장)

- "4×4 부분 마방진 정답을 자동으로 채워주는 프로그램이 필요하다."
- "대각선 검증 함수를 과제 템플릿에 넣어 주면 된다."

### 진짜 문제 (한 문장)

**여러 제약(행·열·대각선)이 동시에 걸린 과제에서, 중간에 "어디까지 맞고 어디가 틀렸는지"를 순서 있게 판별하지 못하면 학생은 "거의 맞음" 착각 후 시간을 쓰고도 제출을 미완료로 끝내고, 같은 유형 과제를 다시 열지 않게 된다.**
